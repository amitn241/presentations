SQL or No SQL ?

That is the question

But what is a No SQL database?

"...a mechanism for storage and retrieval of data that is modeled in means other than the tabular relations used in relational databases"

No SQL means none - relational


  • Do not follow the relational model

  • No tables with flat fixed-column records

  • Self-contained aggregates that may include hierarchical data and BLOBS

  • No need for object-relational mapping

NoSQL Database Types

There are currently over 100 No SQL databases

  • Document databases
  • Key-Value stores
  • Graph stores
  • Wide-column stores



Benefits of No-SQL Databases

  • Dynamic schemas
  • Open source
  • Simple APIs - easier to use
  • Distributed - elastic scaling out
  • Less need for DBA
  • Big data
  • Operating costs
"Database schema is essential to a good data driven application"


Advantages of Dynamic Schemas

  • No data migrations
  • Agile development
  • Less down time
  • Suitable for dynamic programming languages

Simple API

#!/usr/bin/env python
# encoding: utf-8

from datetime import datetime
from pymongo import MongoClient

client = MongoClient()
db = client['my-database']

db.movies.insert({
    'title' : 'Casablanca',
    'director' : 'Michael Curtiz',
    'release' : datetime.strptime('11 26 1942', '%m %d %Y'),
    'actors' : [
        'Humphrey Bogart',
        'Ingrid Bergman',
        'Paul Henreid',
        'Claude Rains',
        'Conrad Veidt',
    ],
    'genres' : [
         'Drama', 'Romance',  'War'
    ]
})

Simple API


  • Simple interfaces to store and query data

  • Low level data manipulation and selection methods

  • Some APIs provide protocols such as HTTP REST with JSON data transport

  • Database as a service

Distributed


  • Scale out, not scale up

  • Some No-SQL DBs can run in a distributed mode which provide auto-sharding

  • Automatic replication and fail-over

  • Integrated caching capabilities

Disadvantages


  • Storage is the primary focus

  • No standards

  • Lack of consistency (Eventual consistency)

  • Lacks in analytics and BI capabilities

When not to use ?


  • You already have legacy code which uses RDBMS

  • You must have ACID transactions

  • None technical reasons require you to use a commercial enterprise DB

When to use?


  • Your application works around a JSON API

  • Your application has an evolving data schema

  • You are starting a new project and you have design freedom

  • You want / need auto-sharding

  • You have high volume traffic

Document Store

Key Principals


  • Documents are self-contained aggregate data structures

  • Documents consist of attributes (key-value pairs)

  • Attributes have data types which can also be nested

MongoDB (from "humongous")



  • Cross platform document-oriented database
  • Been around since 2007
  • Document-oriented database based on binary JSON (BSON)
  • Supports indexing
  • Supports ad-hoc queries
  • Supports replication
  • Supports file storage
  • Scales horizontally
  • Server-side Javascript execution
  • Aggregation and Map Reduce
